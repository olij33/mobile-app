RWW LDP Mobile App with WebCredits 

About

BACKGROUND

Working with W3C Community groups around Read-Write Web.  We’re building W3 Standards Compliant distributed Linked-Data Read-Write Cloud storage system/s.  My frustration with these systems, is that the it is near impossible to show others how they work due to the difficulty in signing-up.

At the moment; to write a card [1] someone needs to type it out in Turtle.  Most people don’t know what turtle is, and the problem continues. The object with this open-source project, is to build something that is useful, easy to use & hopefully popular as to create new opportunities in developing the decentralised web-community, encouraging interest in developing new applications, etc.


FUNCTIONS OF THE APP

The first function is to support the creation of RWW[2]/LDP[3] accounts.  This is done by creating a FOAF Profile in relation to a WebID[4].  A number of these RWW platforms exist, from henry’s RWW-PLAY [5], to Kingsley’s ODS / OpenLink Software [6] and the various solutions built by the team in MIT [7].

The current easy examples are written in PHP [8][9] providing an array of application examples, such as a Calendar [10], Microblogging application [11][12] and continuing examples of address book [13][14] which demonstrates how data stored and retrieved semantically, in a format that can be searched on the web [15] and a file-browser [16]

Beyond the proof of concept work in using these platforms, melvin[17] and ‘webized’[18] Bitcoin [19][20][21] which now needs a method to store the private key, and create transactions between parties.  In-turn, the use-case more particularly, is to facilitate the use of a ‘web credit’ as a stamp, enabling people to use currency (in the form of ‘web credits’) in association to sharing on the web.  The initial outline was posted [22] in july where it is that in essence, i’m seeking the initial development of an app (that supports the USE-CASE 3: WebCredits ) on a mobile device, as to support the storage of a private key.

The theory being that in the first instance - a user may ‘like’ or otherwise create a number of transactions.  they may create a setting that allows these transactions to be verified immediately (upto a certain amount, or amount per day, etc.) or may wait until they goto their phone and authorise transactions at a latter time.  The WebCredits system includes ACL’s (access control) which is controlled via WAC[23].  if a resource is not available until a payment has been made, this can be controlled via WAC.

The underlying premise, is that people will be able to share resources, obtain URI’s for resources to be shared, create a stream of information that might share content with CreativeCommons associations[24][25] or form groups on projects [26][27] where stakeholders contribute, approve, deny and develop new software whilst being recognised for their work in the form of web-credits, which in-turn forms a type of ‘proof of stake’[28] but in relation to a products development.  

At the moment - in the first iteration - i’m simply looking to create a mechanism that does not (yet) allow for the creation of new projects (multi-actor entities) but rather, start with the basics.

So, this would include;

1. A Personal Profile with Addressbook function

1.1 The ability to create friendship groups

1.2 the ability to change what information is displayed to what group (i.e. family might see home phone, business friends might see mobile phone or different email)


2. The ability to support a WebCredits Wallet 

2.1 create a public/private key compatible with the block-header for WebCredits

2.2 the ability to sign a web credit

2.3 the ability to send a web-credit to someone if your address-book

2.4 the ability to attach a web credits request to an asset, or give web credits in relation to an action.  


IE: 

- you can see this picture for $0.0001 web credits 

- i decide i will give you $0.00001 WebCredits when i like your microblog post or $0.0002 if i decide to share it


3. Microblogging

Cimba [11] / kima [12] demonstrate a basic microblogging platform.  Here[29] are some examples of how that data is stored. This [30] gives you some information on how to read/write to the data space.  An example of an application that is being used for Instant messaging (as apposed to posting) is here [31] however i’m not sure atm about whether we’re simply creating different types of messaging; and therefore, perhaps ‘microblogging’ is ‘channel based’ whereas IM is a private type of communication…


4. GIS data

In terms of a format to store location information - here [32], however i feel it’s important that when this data is made public - we provide an option in the address book / friendship groups section to lower the resolution of the GPS data via Freebase [33] or similar.


I feel it’s important to identify where the user is: and store that data, so that if a users android device (i.e.: phone) goes missing, they’re able to identify where it has been if (or when) it’s been doing transactions.  I think it is achievable in the first instance to present an interface to the user that shows them, where they’ve been, etc. However this should be provide data.

TECHNICAL REQUIREMENTS

IMAGE MANAGEMENT:  We’ll need to be able to manage media content.   In the first instance, images - the ability to create a master / derivatives and use settings on the phone to select whether the master is uploaded on 3G or WiFi connections to the phone.

This functionality will be extended to manage other forms of media, such as Audio and Video files. 

CREDENTIALS MANAGEMENT

In the first instance, no e-commerce (meaning a form of national currency instrument) will be used.  However, part of the authentication system will be using a WebID (TLS Certificate) in association to a Persona (FOAF documents).  Persona Documents can be stored on different CloudStorage Servers (i.e.: RWW.io  / Data.fm, etc.) with the same WebID TLS Certificate. 

NOTE:  in latter versions i plan to enable a method to create your own RWW websites (i.e.: http://your.name ) and support credentials https://identus.org/  which in-turn should support http://dig.csail.mit.edu/2010/Papers/IAB-privacy/httpa.pdf and an array of https://web-payments.org/ other similar applications.  i note that the difference relates to international identity related requirements…


TAGS - NFC + 2D Barcodes

I would like to progressively store TAGS (info about ‘things’  linked to 2d barcode / NFC Chip identifiers) in the RWW DataSpace.  These tags will be linked to ontological information about the device, which may be used for authentication purposes, support for the device (i.e. updating bios in tv), event management, etc.
